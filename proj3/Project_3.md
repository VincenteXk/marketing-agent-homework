# Project 3：CRM & Customer Feedback Response

（客户关系管理与反馈处理）

## 项目介绍

在全渠道营销时代，品牌每天面对来自售前咨询、电商评论、社交媒体等多个触点的海量客户反馈。如何快速识别、精准分层并将反馈转化为 CRM 决策依据，是现代品牌运营的核心竞争力。

本项目要求各小组扮演“品牌运营方”，围绕 Project 1–2 中已确立的产品，利用 AI Agent 构建一套数据驱动的客户反馈分析与响应系统，并回答：

> 企业应如何将 Agent 嵌入 CRM 流程，在效率、客户价值与品牌风险之间做出权衡？

## 具体要求

### 1. 跨组随机配对与反馈收集

#### 1.1 随机配对机制

为模拟真实的多来源客户反馈场景，本项目采用跨组随机配对方式：

- 每组接收来自 2 个其他小组的客户反馈（扮演该产品的消费者）。共20条。

#### 1.2 标准化数据格式

所有反馈（无论 Human-generated 还是 Agent-generated）必须整理为如下结构的 Excel 或 CSV 表格：

| 字段名                 | 类型 | 说明                                                |
| ---------------------- | ---- | --------------------------------------------------- |
| `feedback_id`        | 文本 | 唯一编号，如 `G01-001`                            |
| `source_type`        | 分类 | `human` / `agent` / `online`                  |
| `stage`              | 分类 | `pre-purchase`（售前）/ `post-purchase`（售后） |
| `feedback_text`      | 文本 | 反馈原文                                            |
| `customer_profile`   | 文本 | 消费者画像摘要（年龄段、使用场景、购买动机等）      |
| `purchase_frequency` | 数值 | 预期或实际购买频次（次/年）                         |
| `avg_order_value`    | 数值 | 预期或实际客单价（元）                              |
| `channel`            | 文本 | 购买或互动渠道（如电商/线下/社交媒体）              |

各组可在此基础上增加字段（如推荐意愿、是否首次购买等），但需要保留上述核心字段。

#### 1.3 反馈来源要求

最终每组应整合 40–60 条反馈，来源构成建议如下：

- **Human-generated**（来自其他小组）：共约 20 条，其中售前与售后各占一定比例。
- **Agent-generated**（本组自行生成）：共约 20–40 条，必须基于 Project 1 中形成的消费者画像进行有依据的系统化模拟，而非随机生成。请明确说明每条模拟反馈对应的消费者类型及生成逻辑。
- 售前与售后反馈各至少 15 条。

### 2. Agent 响应设计

#### 2.1 售前 vs. 售后的差异化设计

Agent 在售前与售后场景中面临不同挑战，各组需分别说明 Prompt 或工作流如何适配两类场景：

- **售前反馈**：重点在信息解释、消除顾虑与促成转化；Agent 是否更擅长标准化问题解答？哪些咨询需要个性化引导？
- **售后反馈**：重点在情绪修复、问题解决与关系挽留；Agent 是否更擅长处理程序性投诉而非情感性投诉？

#### 2.2 每条反馈的标准输出

针对结构化数据表中的每条反馈，Agent 至少需要输出：

1. 问题归类与情绪判断。
2. 客户价值层级与优先级评分。
3. 建议回复草案（符合品牌调性）。
4. 建议处理方式（如标准回复 / 补偿 / 退换货 / 升级人工）。
5. 是否建议人工介入，并注明理由。

#### 2.3 差异化响应策略

各组必须明确制定响应策略与客户价值层级、反馈阶段和风险等级的规则，例如：

- 高价值 + 高风险：专属客服 + 人工介入，响应时限 `<= 2h`。
- 中价值 + 中风险：Agent 辅助 + 人工审核，响应时限 `<= 24h`。
- 低价值 + 低风险：Agent 全自动回复，响应时限 `<= 48h`。

具体规则由各组根据产品特点自行设计，但必须明确说明分层依据。

#### 2.4 Human vs. Agent 小规模对照

为控制工作量，本项目不要求对所有反馈进行人工回复。请从高价值、高风险或高歧义样本中抽取 6–10 条，在不参考 Agent 输出的前提下独立完成人工回复，并从以下维度进行对比：

- 情感共鸣度。
- 信息准确性与品牌一致性。
- 潜在品牌风险（是否有过度承诺、语调失当等）。
- 可扩展性（该回复是否可以模板化）。

可邀请组外同学进行盲测评分，以增强对比的客观性。

### 3. 客户价值测算与分层

本节是项目的核心量化模块，旨在回答：

> 面对同样的反馈，企业为何不应对所有客户投入相同资源？

#### 3.1 简化 CLV 估算

各组需基于结构化数据表中的字段，对每位客户或客户类型估算简化客户生命周期价值。可采用课件上的公式或如下公式：

`CLV ≈ 客单价 × 购买频次/年 × 预计留存年限 × 毛利率`

字段对应建议：

- `avg_order_value`：客单价
- `purchase_frequency`：购买频次
- 预计留存年限：由画像估算
- 毛利率：由产品设定

若属于新产品或缺乏历史数据，可采用代理评分方法（须明确打分规则），例如综合“预期购买频次 × 客单价 × 推荐潜力”构建相对价值指数。

#### 3.2 RFM 评分（或类 RFM）

若产品具备交易数据，请定义 Recency、Frequency、Monetary 的具体含义并打分。若属于新产品 / 无历史数据，可采用类 RFM 方法，例如：

- `R`：最近一次互动时间或购买意向紧迫度。
- `F`：预期购买频次。
- `M`：预期客单价或 CLV 估算值。

请说明为何该替代方式在当前情境下合理。

#### 3.3 客户分层

基于 CLV 或 RFM，将客户划分为至少 3 个层级，并在报告中呈现：

- 每层的判定规则与客户数量。
- 每层的核心特征与管理含义。
- 可视化（散点图、分层矩阵或分布图）。

#### 3.4 综合优先级评分

在客户价值的基础上，建议引入以下维度构建综合优先级评分（Priority Score）：

`Priority Score = w1 · CLV 分位 + w2 · 情绪强度 + w3 · 风险等级 + w4 · 紧急程度`

各组可自行定义权重 `wi` 及各维度取值方式，但须说明权重逻辑的依据，并据此对所有反馈排序，确定响应优先级。

### 4. 讨论与管理反思

本部分要求各组基于前述数据与分析，围绕以下三个核心问题展开有理有据的管理层讨论，而非泛泛的心得总结：

#### Q1. Agent 的适用边界在哪里？

基于售前与售后的对比分析，说明哪些任务适合 Agent 自动处理，哪些必须人工主导。请结合以下四类因素给出判断，并以你们的数据样本为证据支撑：

- 任务标准化程度
- 客户价值层级
- 情绪强度
- 潜在品牌风险

#### Q2. 资源有限时，优先策略如何确定？

结合你们的优先级评分结果，说明若客服资源受限，企业应优先响应哪类客户与反馈。“按价值优先”与“按情绪紧急程度优先”两种策略在你们的数据中会产生哪些不同结果？哪种方式更符合你们产品的 CRM 目标？

#### Q3. 若这些回复被真实客户看到，品牌会承担什么风险？

从你们 Human vs. Agent 对照的样本出发，识别 Agent 回复中最可能出现的风险点（如信息误导、情感失当、过度承诺、品牌语调不一致等），并提出至少两项可落地的治理机制（如人工抽检规则、升级阈值、监控指标等）。

## 提交要求

1. **PPT / PDF**：完整呈现反馈数据概述、客户价值分析与分层结果、Agent 系统输出展示、Human vs. Agent 对照、三个讨论问题的回答。
2. **结构化数据文件（Excel / CSV）**：原始反馈表（含所有核心字段）及 Agent 输出结果表。
3. **Agent 信息**：提供截图及 Prompt 核心逻辑说明。

请将所有文件分别作为附件提交，无需压缩。报告中须清楚区分哪些内容来自真实收集、哪些来自 Agent 模拟；凡涉及评分或估算的部分，必须说明规则依据。
